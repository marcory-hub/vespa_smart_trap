## Script-by-script overview

### `scripts/experiment_set_match/run_experiment.py`

- **Role:** End-to-end GV2 slideshow experiment runner that starts the slider server, captures serial `INVOKE` events, aligns them to image frame windows, computes set-based metrics, and exports reports.
- **Runtime dependencies:**
  - Python stdlib (`argparse`, `csv`, `json`, `threading`, `subprocess`, `webbrowser`, etc.).
  - Third-party: `pyserial` (`import serial`).
  - External process: launches `scripts/image_slider_web/server.py`.
- **Primary inputs:**
  - `data/test/images/*.jpg`
  - `data/test/labels/*.txt`
  - Serial stream from `/dev/tty.usbmodem*` (default glob).
  - Slider run log from `scripts/image_slider_web/outputs/<run_id>/run_log.csv`.
- **Primary outputs:** creates `outputs/<timestamp>__<model_name>/` with:
  - `run_config.json`
  - `per_image_results.csv`
  - `review_candidates.csv`
  - `per_class_metrics.csv`
  - `raw_invoke_events.jsonl`
  - `unmatched_invoke_events.jsonl`
  - `parse_errors.json`
  - `slider_server.log`
- **Current limitations/risks:**
  - Picks the first serial device match automatically, which may bind to the wrong device if multiple ports match.
  - Uses `python3` for subprocess launch instead of the active interpreter, which can drift from the current virtual environment.
  - The wait for frame completion uses a fixed 1200-second ceiling that is not exposed as a CLI parameter.

### `scripts/image_slider_web/server.py`

- **Role:** Local HTTP server for slideshow runs. Serves static UI, image manifest, image bytes, run start endpoint, and CSV logging endpoint.
- **Runtime dependencies:** Python stdlib only (`http.server`, `argparse`, `csv`, `json`, etc.).
- **Primary inputs:**
  - Static UI files under `scripts/image_slider_web/static/`.
  - Image files from `data/test/images/*.jpg`.
  - Client log POST payloads for shown images.
- **Primary outputs:**
  - `scripts/image_slider_web/outputs/<yyyy-mm-dd_HHMMSS>/run_log.csv`.
  - HTTP JSON responses for `/manifest.json`, `/start`, and `/log`.
- **Current limitations/risks:**
  - Global run state is mutable shared process state with no explicit lock, while the server runs as `ThreadingHTTPServer`.
  - `GET /start` has side effects (run directory creation), so page reloads and retries can create extra run folders.
  - Manifest shuffle seed is fixed to 42 in code and cannot be overridden from CLI.

### `scripts/image_slider_web/static/app.js`

- **Role:** Browser slideshow controller. Starts a run, loads manifest, presents calibration + 2-second image cadence, and logs each displayed image.
- **Runtime dependencies:** Browser runtime with `fetch` and keyboard event handling.
- **Primary inputs:**
  - `/start` run metadata from server.
  - `/manifest.json` list of images.
  - Keyboard controls (`Space`, `ArrowRight`, `R`).
- **Primary outputs:**
  - Repeated POSTs to `/log` with `{index, filename, viewpoint, species, shown_at_iso}`.
  - In-browser display state (overlay, calibration, current image).
- **Current limitations/risks:**
  - Intro duration (`20s`) and frame duration (`2s`) are hardcoded in client code with no run-time configuration.
  - Manual `ArrowRight` stepping can introduce cadence deviations while still writing log rows.
  - Restart behavior uses a short delayed re-entry and shared mutable state (`stopping`, `paused`), which is harder to reason about under rapid key presses.

### `scripts/gv2_yolomodel_test/swift_yolo_output.py`

- **Role:** Minimal serial helper for sending `AT+INVOKE` and printing incoming serial text.
- **Runtime dependencies:** `pyserial`.
- **Primary inputs:**
  - Hardcoded serial port path (`/dev/tty.usbmodem58FA1047631`).
  - GV2 serial output lines.
- **Primary outputs:**
  - Writes `AT+INVOKE=-1,0,1` to serial.
  - Prints decoded serial stream to stdout.
- **Current limitations/risks:**
  - Hardcoded serial path reduces portability and reproducibility.
  - Sends the same invoke command twice (once before readiness loop and again after).
  - No structured logging, timeout strategy, or controlled shutdown path.

### `scripts/image_view_sorter/sort_views.py`

- **Role:** Interactive local image sorter that opens each image and moves it into `top_down`, `side`, or `other`.
- **Runtime dependencies:** Python stdlib (`os`, `shutil`, `pathlib`) and macOS `open` command.
- **Primary inputs:**
  - Files in `/Users/md/Developer/vespa_smart_trap/test/images` with image extensions.
  - Operator keyboard choices (`t`, `s`, `o`, `q`).
- **Primary outputs:**
  - Moves files into:
    - `/Users/md/Developer/vespa_smart_trap/test/images/top_down`
    - `/Users/md/Developer/vespa_smart_trap/test/images/side`
    - `/Users/md/Developer/vespa_smart_trap/test/images/other`
- **Current limitations/risks:**
  - Uses an absolute path under `test/images`, while experiment scripts use `data/test/images`.
  - Uses `os.system('open "...")` with shell interpolation.
  - Performs move operations without dry-run mode or transaction log.

## Improvements-only findings (actionable)

### Critical impact

1. **Path mismatch can break dataset flow**
   - **Affected path:** `scripts/image_view_sorter/sort_views.py`
   - **Risk:** Sorting can affect a different tree (`test/images`) than the tree used by the experiment (`data/test/images`), causing stale inputs or invisible data edits.
   - **Recommendation:** Add a required `--image-dir` CLI argument and align default to `data/test/images` (or fail if both trees exist and differ).

2. **Implicit serial port selection can target wrong hardware**
   - **Affected path:** `scripts/experiment_set_match/run_experiment.py`
   - **Risk:** First-match port selection can silently capture from a non-GV2 device when multiple serial devices are connected.
   - **Recommendation:** Require explicit `--serial-port`, or show discovered ports and require user confirmation before run start.

### High impact

3. **Environment drift risk when launching slider subprocess**
   - **Affected path:** `scripts/experiment_set_match/run_experiment.py`
   - **Risk:** Calling `python3` directly may run a different interpreter than the current environment used for the main script.
   - **Recommendation:** Launch subprocess with `sys.executable` to guarantee interpreter parity.

4. **Threaded server state has no explicit synchronization**
   - **Affected path:** `scripts/image_slider_web/server.py`
   - **Risk:** Concurrent `/start` and `/log` operations can race on mutable global state in threaded mode.
   - **Recommendation:** Add a lock around run state transitions and log writes, or switch to single-thread server mode if concurrency is not required.

5. **Hardcoded device port limits repeatability across machines**
   - **Affected path:** `scripts/gv2_yolomodel_test/swift_yolo_output.py`
   - **Risk:** Script fails immediately outside one host/device mapping.
   - **Recommendation:** Add CLI flags (`--serial-port`, `--baudrate`, `--timeout`) and optional glob discovery fallback.

### Medium impact

6. **Run timing is partly fixed in code, reducing configurable reproducibility**
   - **Affected paths:** `scripts/experiment_set_match/run_experiment.py`, `scripts/image_slider_web/static/app.js`
   - **Risk:** Intro/frame timing and wait ceilings are split between Python and JS constants, making controlled run variants difficult.
   - **Recommendation:** Centralize timing config in server-provided JSON and persist it in `run_config.json`.

7. **Side-effectful GET endpoint creates accidental run folders**
   - **Affected path:** `scripts/image_slider_web/server.py`
   - **Risk:** Browser refreshes or tooling prefetches can create runs unintentionally.
   - **Recommendation:** Change run creation to POST `/start` and keep GET endpoints idempotent.

8. **Manual stepping can invalidate strict cadence assumptions**
   - **Affected path:** `scripts/image_slider_web/static/app.js`
   - **Risk:** `ArrowRight` fast-forward inserts extra display/log events outside constant 2-second cadence assumptions.
   - **Recommendation:** Log explicit event type (`auto` vs `manual_step`) and optionally disable manual stepping during benchmark runs.

9. **Legacy helper script duplicates invoke command without explicit intent**
   - **Affected path:** `scripts/gv2_yolomodel_test/swift_yolo_output.py`
   - **Risk:** Duplicate `AT+INVOKE` may produce unexpected device behavior depending on firmware command handling.
   - **Recommendation:** Send invoke once after readiness confirmation, and document the startup contract in the script docstring.
